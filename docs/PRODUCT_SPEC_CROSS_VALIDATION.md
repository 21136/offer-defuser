## 一、P0 意见验证（R-001～R-004）

### R-001 步骤库数据模型与 §2.3 生成逻辑矛盾

- **判定**: 部分采纳
- **理由**: §2.3「匹配人格 + 场景 → 每步 7 选 1」与附录 `StepLibrary.personas: Record<string, StepTemplate[]>` **语义一致**（均为每人格 8 步 × 每步 7 选项），Week 2 亦写明「16 人格 × 8 步 × 7 条」。真正缺口是**未在正文选定并写死方案 A/B**，多 AI 并行时可能有人实现「场景共用骨架 + 人格修饰」。产品承诺「人格匹配话术池」且 16 类人格需肉眼可辨，应锁定**方案 A（全量分叉）**。
- **对 PRODUCT_SPEC 的修改建议**:
  1. 在 §2.3 增加「数据模型定稿」小节，明示采用方案 A，并贴出与附录一致的 `StepLibrary` 定义。
  2. 将生成逻辑改写为：

     ```
     category = resolveJobCategory(jobInput)
     templates = steps[category].personas[personaId]  // 固定 8 个 StepTemplate
     plan.steps[i] = weightedRandom(templates[i].options) + interpolate(vars)
     ```

  3. 删除或标注方案 B 为「未来优化，v1 不采用」，避免双轨实现。

---

### R-002 模块 A → 模块 B 串联规则未定义

- **判定**: 采纳
- **理由**: §5.3 仅写 `DefuserPage ← personaType + jobInput`，未定义 `NO_PERSONA` / `HAS_PERSONA` 行为。拆解器核心价值是「人格匹配话术」；无规则时实现方可能硬拦（流失）或静默用错人格（损害可解释性）。传播上 ResultPage 应有明确出口串联 B 模块。
- **对 PRODUCT_SPEC 的修改建议**:
  1. 新增 **§5.4 用户状态与模块串联**：

     | 状态 | 条件 | DefuserPage 行为 |
     |------|------|------------------|
     | `NO_PERSONA` | 无 `personaId` | 允许进入；使用 `DEFAULT_PERSONA_ID`（建议人格 #2「简历空白焦虑者」或单独 `unknown`）；顶栏 Banner「测 2 分钟，拆解更准」→ `/test` |
     | `HAS_PERSONA` | localStorage 有 `personaId` + `scoreVector` | 自动带入；页头「为【{人格名}】定制」 |

  2. ResultPage 主 CTA：「用这个人格拆解我的 Offer」→ `/defuser?persona={id}`。
  3. **不强制先测后用**（与 Q-02 一致），但 UI 上持续引导补测。

---

### R-003 缺少可分享的结果 URL（传播链路缺口）

- **判定**: 采纳
- **理由**: §九传播路径对标 SBTI，但 §5.2 仅有 `/result` 无 query；结果仅存 localStorage，好友无法通过链接复现。截图是微信主路径，但**链接二次传播**是 SBTI 类产品的标准能力，且与零后端不冲突（query 解码即可）。属传播链路 P0 缺口。
- **对 PRODUCT_SPEC 的修改建议**:
  1. 新增 **§5.5 分享与深链**：

     ```
     GET /result?p={personaId}&v={delay},{apply},{perfect},{interview}  // v 可选
     ```

  2. 行为：`p` 合法 → 渲染人格解读；有 `v` → 渲染雷达图；缺 `p` 或校验失败 → 友好降级首页。
  3. §四 P0 功能表增加一行：**可分享结果 URL**。
  4. §5.3 数据流在 ResultPage 后增加：`encodeShareUrl()` → 复制链接 / 海报二维码（二维码可标 P1）。

---

### R-004 8 步生成应区分「步骤类型」与「文案随机」

- **判定**: 采纳
- **理由**: §2.3 只写「每步随机 7 选 1」，未强调 `position` 为**固定语义槽位**。附录 `StepTemplate.position` + `title` 已暗示槽位固定，但正文未写死，存在实现成「8 步标题也随机」的风险，会直接破坏「可执行行动方案」承诺。`deadline` 作为插值变量在 §3.3 有示例但未与步骤生成挂钩。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §2.3 明确：**步骤语义由 `position` 决定，不可打乱；仅槽位内文案加权随机**。
  2. 固定八步槽位表：

     | position | title | 意图 |
     |----------|-------|------|
     | 1 | 搞清门槛 | JD 拆解 / 硬性条件 |
     | 2 | 改简历 | 针对该岗改一版 |
     | 3 | 去投递 | 今天投出去 |
     | 4 | 笔试准备 | 客观题 / 手撕 |
     | 5 | 面试准备 | 项目话术 |
     | 6 | 模拟练习 | 自问自答 / 录音 |
     | 7 | 跟进复盘 | 邮件 / 内推追问 |
     | 8 | 心态复位 | 防内耗收尾 |

  3. `stepGenerator` 输入增加 `daysLeft`（由截止日计算），写入 `interpolate` 变量集。
  4. 注明：`JobCategory` 可覆写单步 `title`（如国企弱化「海投」措辞），但 `position` 序列不变。

## 二、P1 意见验证（R-005～R-010）

### R-005 欧式距离匹配的分布与可解释性

- **判定**: 部分采纳
- **理由**: 欧式距离 + 16 点聚类不均是真实风险；§3.1 已写 optional Top-3，但缺**分布验收**与**结果页可解释性**规范。Top-2 匹配展示（如「68% A + 32% B」）比裸 Top-3 列表更易传播，但需定义 `similarity` 计算公式（附录有字段未定义算法）。维度加权距离属增强项，非 v1 阻塞。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §3.1 增加 **分布验收**：构建脚本模拟 ≥10k 随机答卷，输出 16 人格直方图；任一型占比 >35% 或 <3% 时调整 `personas.json` 标准向量。
  2. 将 Top-3 从 optional 升为 **P0 结果展示**：主结果 +「次人格」；`similarity = max(0, 1 - distance / DISTANCE_SCALE) * 100`，`DISTANCE_SCALE` 写死常量。
  3. 结果页展示 **Top-2 匹配条**（主 + 次一人格及百分比）。
  4. 维度加权 `w1..w4` 记入 §3.1「P2 可选」，v1 不加权。

### R-006 归一化公式需在 spec 中写死

- **判定**: 采纳
- **理由**: §2.1「±N 分后归一化到 [0,100]」无公式，多 AI 实现 `scoreCalculator` 极易不一致（累加后 clamp？按理论 min/max 线性映射？）。附录 `Partial<ScoreVector>` 也说明每题各维增量不同，**必须预计算 bounds**。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §2.1 / §3.1 补充伪代码：

     ```typescript
     // 构建期从 questions.json 导出
     rawMin[d], rawMax[d] = 该维 12 题所有选项累计的最小/最大可能值
     normalized[d] = clamp((raw[d] - rawMin[d]) / (rawMax[d] - rawMin[d]) * 100, 0, 100)
     ```

  2. 新增构建脚本：`scripts/genScoreBounds.ts` → 输出 `src/constants/scoreBounds.ts`。
  3. Week 1 任务清单增加「归一化常量生成脚本 + 单测」。
  4. **保持 12 题**（见 Q-04），不改题量。

### R-007 海报信息架构：单张 9:16 装不下全部内容

- **判定**: 部分采纳
- **理由**: P0 要求单张 9:16 含雷达 + 人格 + **8 步全文** + Slogan，1080×1920 上字号必然过小，与 §九「截图即内容」冲突。拆两种海报合理，但 P0 传播主战场应是**人格结果海报**，8 步全文更适合拆解器页内或第二张海报。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §四 P0 将 Canvas 海报拆为：
     - **`poster-result`（P0）**：人格名 + 雷达 + 标签梗 + Slogan + 短链/二维码位。
     - **`poster-plan`（P1）**：人格 + 8 步精简（每步一行，或摘要 3 条 +「扫码看全文」）。
  2. 原「8 步清单上海报」从 P0 移至 P1 或改为「Top 3 步骤摘要」。
  3. ShareCanvas 组件注明两套模板 ID，导出 API 统一 `exportPoster(type)`。

### R-008 微信 H5 与链接预览

- **判定**: 部分采纳
- **理由**: §四提微信 H5、§九讲裂变，但未写分享形态优先级。纯 SPA 无 SSR 时，微信链接卡片 og 体验确实差；**长按保存海报**才是微信生态主路径，spec 应写明以免开发过度投入 prerender。静态默认 og + 动态 `document.title` 成本低、收益中等。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §九新增「微信分享优先级」：
     - **首选**：海报长按保存 → 朋友圈/小红书。
     - **次选**：复制结果链接（R-003）。
     - **辅助**：`index.html` 默认 `og:title` / `og:image`；ResultPage 设置 `document.title = 你是【{人格名}】| Offer拆弹专家`。
  2. 明确 **不做 SSR/prerender**（与零后端一致），除非后续单独立项。
  3. P1 可考虑结果页静态 fallback 图（Canvas 生成后不上传，仅客户端预览用）。

### R-009 娱乐性免责声明

- **判定**: 采纳
- **理由**: 「求职内耗人格」+ 16 类梗命名易被理解为心理测评；作品集场景也存在「中性别名」需求（见 Q-06）。免责声明成本低，合规与产品诚实边界必要。
- **对 PRODUCT_SPEC 的修改建议**:

     > 本测试基于娱乐化模型，仅供求职心态自嘲与参考，不构成心理或职业咨询。

  2. ResultPage / 海报底部均展示（海报可用小字）。
  3. Q-06 的「面试演示版」单独记 P2，主站保持梗名。

### R-010 字段命名 `deliver` 易歧义

- **判定**: 部分采纳
- **理由**: `deliver` 在英文语境易联想到 delivery/交付，与 D2「投递策略」不符；但 spec 仍为初稿、附录与 §2.1 统一使用 `deliver`，**尚未量产 JSON**。更稳妥是 spec 定稿前改名并全文统一，而非仅加注释。
- **对 PRODUCT_SPEC 的修改建议**:
  1. 将 `ScoreVector.deliver` 重命名为 **`apply`**（全文统一，含分享 URL `v=` 四维顺序）。
  2. 在 `Dimension` 类型加注释：`apply — application strategy (投递策略)`。

- **理由**: 「轻埋点」未定义实现边界，Dashboard 若展示「测试次数」易被误解为全站 UV。零后端下应写清数据仅本机。

  1. §四 P1「数据看板」描述改为：**本机统计**（localStorage 聚合）。

     ```
     轻埋点 = localStorage 事件计数 + 可选第三方脚本（Umami/Plausible，无 cookie 方案）

     ```

  3. DashboardPage UI 文案：「本机数据，换设备不共享」。

### R-012 暗色模式与海报一致性

- **判定**: 采纳
- **理由**: P1 暗色模式 + P0 海报若跟随主题，ShareCanvas 需双倍绘制逻辑且导出可能在暗色下可读性差。海报作为**对外传播资产**应固定亮色品牌稿。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §四 P1 暗色模式下增加约束：**海报 Canvas 固定亮色主题**，不读取 `appStore.theme`。
  2. ShareCanvas 规格：`theme` 参数仅 App 预览用，**导出恒为亮色**。

### R-013 素材生产规范（多 AI 并行必备）

- **判定**: 采纳
- **理由**: 多 AI 并行生产步骤文案时，缺统一规范会导致语气漂移、`{{变量}}` 不一致、同槽位 7 条语义重复。这是**质量/一致性治理**，不是体量否决。
- **对 PRODUCT_SPEC 的修改建议**:
  1. §二末或 §七增加引用：新建 `docs/CONTENT_STYLE_GUIDE.md`。
  2. 规范至少包含：

     | 规范项 | 说明 |
     |--------|------|
     | 语气 | 松弛自嘲 / 禁止说教 / 单条 ≤ 80 字 |
     | 变量 | 统一 `{{jobName}}` `{{deadline}}` `{{daysLeft}}` |
     | 人格差异 | 每人格至少 3 个「禁用词」+ 3 个「口头禅」 |
     | 去重 | 同场景 7 选项语义不可重复，仅措辞不同 |
     | Review | 合并前跑 JSON Schema 校验 + 重复率脚本 |

  3. §5.1 `assets/` 下注明：合并前跑 `npm run validate:content`。

- **对 PRODUCT_SPEC 的修改建议**:

     | questions.json | 12 题 × 4 选项 | 含分数映射 |

     | personaTexts.json | 16 套解读 | 名称/优劣势/建议 |
     | steps/frontend.json | 16 × 8 × 7 | 方案 A 全量分叉 |
     | steps/general.json | 16 × 8 × 7 | 同上 |
     | steps/state-owned.json | 16 × 8 × 7 | P1 上线 |
     | **步骤选项合计** | **2,688** | 步骤选项为主力 |

  2. 附并行拆分示例：`AI-1 人格 1–4 + frontend`，`AI-2 人格 5–8 + general` 等。
  3. 验收标准：每文件通过 Schema + 槽位 title 与 §2.3 八步表一致。

### R-015 E2E 测试范围

- **判定**: 部分采纳
- **理由**: 三条黄金路径合理，但第三条「分享 URL 解码」依赖 R-003 合入后才有意义；「换一句」应断言**仅重随机当前槽位、其他步不变**（R-004 的回归点）。不宜扩太多 E2E，但应写进 spec。

  2. 其余 P2 页面（图鉴、看板）不做 E2E 阻塞。

## 四、开放问题定论（Q-01～Q-06）

### Q-01 步骤库用方案 A 全量分叉还是方案 B 骨架+修饰？

- **判定**: 采纳（选 **方案 A**）
- **理由**: 产品定位「人格匹配话术池」、16 类人格梗名差异化、Week 2 已按全量分叉排期；方案 B 降低维护但「肉眼可见差异」弱，与 SBTI 式「我是哪一种」传播不符。
- **修改建议**: §2.3 标题改为「层级 3：分场景 × 分人格步骤话术池（方案 A）」；详见 R-001。

---

### Q-02 是否强制先测后用？

- **判定**: 采纳（**不强制**）
- **理由**: 拆解器有独立工具价值；强制测试提高完成成本，小红书链路也可能是「先拆 Offer 再好奇测人格」。应用 Banner + DEFAULT_PERSONA 引导，而非路由拦截。
- **修改建议**: HomePage 两入口均保留；Defuser 内软引导；详见 R-002。

---

### Q-03 分享 URL 是否携带完整四维向量？

- **判定**: 部分采纳（**personaId 必带，向量可选**）
- **理由**: 仅 `p` 可复现人格解读与梗文案，满足主传播；带 `v` 可复现雷达图，增强「像我的」体感，但非必需且略增隐私顾虑（虽非敏感）。建议 **v optional**：有则画雷达，无则展示人格默认示意雷达或隐藏雷达。
- **修改建议**: §5.5 写明 `p` required、`v` optional；复制链接默认仅 `?p=`，「带雷达分享」可选带 `v`。

---

### Q-04 12 题是否增加到 16/20 题？

- **判定**: 采纳（**保持 12 题**）
- **理由**: 快消测试完成率优先；12 题 × 4 维已可支撑 16 型匹配，题量增加对分布改善边际递减，却直接伤害传播漏斗。
- **修改建议**: §2.1 增加：「v1 固定 12 题，题量增加列入 P2 实验。」

---

### Q-05 国企场景是否在 P0 上线？

- **判定**: 部分采纳（**P0：frontend + general；P1：state-owned**）
- **理由**: §2.3 列三场景，但 P0 核心是链路跑通 + 传播；国企文案调性差异大，可并行生产但不阻塞上线。
- **修改建议**:
  1. §2.3 场景表增加「上线阶段」列。
  2. DefuserPage 岗位类型：P0 仅两选项，P1 开放第三项。
  3. Week 2 先 `frontend` 全量跑通联调，另两场景并行填充。
