## 三、P0 意见（建议在编码前修订规格）

### R-001 步骤库数据模型与 §2.3 生成逻辑矛盾

| 项 | 内容 |
|----|------|
| **位置** | §2.3 vs 附录 `StepLibrary` |
| **问题** | §2.3 描述为「匹配人格 + 场景 → 每步 7 选 1」；附录定义为 `personas: Record<string, StepTemplate[]>`，暗示每人格一套完整 8 步 |
| **风险** | 不同开发者/AI 会实现两套不同架构，素材生产规范无法统一 |
| **建议** | 在 spec 中**明确选定一种**，并全文统一 |

**方案 A — 全量分叉（与原 spec 体量一致，适合多 AI 并行）**

```typescript
interface StepLibrary {
  category: JobCategory;
  personas: Record<PersonaId, StepTemplate[]>; // 每人格 8 步 × 每步 7 选项
}
```

**方案 B — 骨架 + 修饰（降低重复，仍可按人格差异化）**

```typescript
interface StepLibrary {
  category: JobCategory;
  skeleton: StepTemplate[];              // 8 步骨架，全员共用
  personaModifiers: Record<PersonaId, PersonaTonePack>; // 语气/禁忌/强调点
}
```

**验证问题**:

1. 产品目标是「16 种人格拆出来的方案差异要肉眼可见」还是「同一骨架不同语气」？
2. 若选方案 A，是否需要定义「人格间最低差异度」验收标准？

### R-002 模块 A → 模块 B 串联规则未定义

| 项 | 内容 |
|----|------|
| **位置** | §1.2、§5.3 |
| **问题** | 数据流写了 `DefuserPage ← personaType + jobInput`，但未规定：未测人格时能否用拆解器？默认人格是什么？ |
| **风险** | 用户跳过测试直进拆解器，B 模块失去「人格匹配话术池」的核心价值 |
| **建议** | 补充 **用户状态机** |

```
状态:
  NO_PERSONA  — 未完成测试
  HAS_PERSONA — 已完成测试，localStorage 有 personaId + scoreVector

规则:
  DefuserPage 入口:
    - HAS_PERSONA → 自动带入 personaId，页头展示「为【xxx】定制」
    - NO_PERSONA  → 允许进入，使用 DEFAULT_PERSONA_ID（建议: "general-anxious" 或单独定义第 17 号「未测默认」）
                    顶部 Banner: 「测 2 分钟，拆解更准」→ 链到 /test

  ResultPage 出口:
    - 主 CTA: 「用这个人格拆解我的 Offer」→ /defuser?persona={id}
```

**验证问题**: 是否应强制「先测后用」以提高传播？还是开放拆解器降低流失？

---

### R-003 缺少可分享的结果 URL（传播链路缺口）

| 项 | 内容 |
|----|------|
| **位置** | §九、§5.2 |
| **问题** | 传播策略对标 SBTI，但路由仅有 `/result`，无编码参数；结果仅存 localStorage，好友无法打开同一份结果 |
| **风险** | 裂变依赖「截图」单一路径，链接二次传播弱 |
| **建议** | 增加 **零后端分享编码**

```
/result?p={personaId}&v={delay},{deliver},{perfect},{interview}
或
/result?r={base64url(JSON)}
```

- 解码失败 → 友好降级到首页
- ResultPage 根据 query 渲染，无需 localStorage
- 海报 CTA 可印短链或二维码（P1）

**验证问题**: query 暴露分数是否影响隐私？是否需要只分享 personaId 不分享向量？

---

### R-004 8 步生成应区分「步骤类型」与「文案随机」

| 项 | 内容 |
|----|------|
| **位置** | §2.3、§3.2 |
| **问题** | 「每步随机 7 选 1」若连步骤语义一起随机，可能产出逻辑跳跃的方案（如先面试后改简历） |
| **风险** | 「可执行行动方案」的产品承诺受损 |
| **建议** | 规格中明确：

```
StepTemplate.position (1-8) → 固定语义槽位，不可打乱
  例: 1调研 2简历 3投递 4笔试 5面试准备 6模拟 7跟进 8心态

每槽位内 7 条文案 → weightedRandom 抽取
deadline 天数 → 作为 interpolate 变量影响紧迫度措辞
```

**验证问题**: 8 步骨架是否应因 `JobCategory` 不同而不同（如国企少「海投」步）？

### R-006 归一化公式需在 spec 中写死

| 项 | 内容 |
|----|------|
| **位置** | §2.1、§3.1 |
| **问题** | 「各选项 ±N 分，12 题后归一化到 [0,100]」未给出公式，AI 并行实现可能不一致 |
| **建议** | 补充伪代码，例如：

```typescript
// 每维原始分 raw[d] ∈ [rawMin, rawMax]，线性映射到 [0, 100]
normalized[d] = clamp((raw[d] - rawMin[d]) / (rawMax[d] - rawMin[d]) * 100, 0, 100)
// rawMin/rawMax 由 12 题选项分数预计算（构建时脚本生成 constants）
```

**验证问题**: 是否改用「每题 0-3 分」等更简单刻度？

| 模板 | 内容 | 优先级 |

| `poster-result` | 人格名 + 雷达 + 标签梗 + Slogan | P0 传播 |
| `poster-plan` | 人格 + 8 步精简（每步一行）| P1 自用 |

**验证问题**: P0 是否坚持单海报？若坚持，8 步是否改为「摘要 3 条」？

- 海报 Canvas 建议固定亮色主题，App 暗色模式不改变海报导出样式，避免双倍绘制逻辑

### R-013 素材生产规范（多 AI 并行必备）

建议新增 `docs/CONTENT_STYLE_GUIDE.md`，至少包含：

| 规范项 | 说明 |
|--------|------|
| 语气 | 松弛自嘲 / 禁止说教 / 单条 ≤ 80 字 |
| 变量 | 统一 `{{jobName}}` `{{deadline}}` `{{daysLeft}}` |
