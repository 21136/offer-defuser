# Offer拆弹专家 — 素材生产规范

> **用途**: 多 AI 并行生产步骤文案时的统一规范，确保 16 人格 × 8 步骤 × 7 选项的文案风格一致、语义不重复、变量使用正确。

---

## 一、变量约定

所有 `StepOption.text` 必须自然嵌入以下变量（模板引擎 `{{key}}` 替换）：

| 变量 | 类型 | 示例值 | 说明 |
|------|------|--------|------|
| `{{jobName}}` | string | "字节前端实习" | 用户输入的岗位名 |
| `{{deadline}}` | string | "2026-07-15" | 用户选择的截止日 ISO |
| `{{daysLeft}}` | number | 20 | 距离截止日的天数（自动计算） |
| `{{personaName}}` | string | "海投摆烂吗喽" | 当前用户的人格梗名 |

### 变量使用规则

1. **每条 option.text 至少包含 1 个变量**（推荐 2-3 个），使文案读起来是个性化的
2. **`{{jobName}}` 和 `{{daysLeft}}` 必须出现**在至少 5/7 条选项中
3. **`{{personaName}}` 可选**，当需要称呼用户时使用（如"作为{{personaName}}的你..."）
4. 变量位置多样化 — 不要全放在句首或句尾
5. 变量后紧跟中文标点无需空格

### ✅ 正确示例
```
"打开{{jobName}}的JD，先圈出3个核心技术要求，{{daysLeft}}天足够你针对性准备"
"{{personaName}}，距{{deadline}}还有{{daysLeft}}天，今天先搞定第一步"
```

### ❌ 错误示例
```
"{{jobName}} {{deadline}} {{daysLeft}} {{personaName}}"  // 纯变量拼凑，无语义
"建议你好好准备面试"  // 无任何变量，不个性化
```

## 三、8 步语义槽位规范

### 槽位 1：搞清门槛
**意图**: 拆解 JD，明确硬性条件，知道自己缺什么
**关键动作**: 打开 JD → 标注"必须"vs"加分" → 自我评估 → 列差距清单
**示例措辞**:
- 随性: "打开{{jobName}}的JD扫一眼，关注【必须】那栏就行"
- 条理: "第1步：拆解{{jobName}}JD，用表格列出硬性条件和加分项"
- 焦虑: "别被JD吓到，{{jobName}}要的东西其实你大半都有"

### 槽位 2：改简历
**意图**: 针对岗位修改简历，关键词对齐
**关键动作**: 对照 JD 关键词 → 修改简历对应段落 → 量化成果
**通用约束**: 最多改 30 分钟，不陷入无限修改

### 槽位 3：去投递
**意图**: 今天完成投递动作
**关键动作**: 选渠道（官网/BOSS/内推）→ 检查附件 → 点击发送
**通用约束**: 必须强调"今天就投"

### 槽位 4：笔试准备
**意图**: 按岗位类型准备笔试/技术面
**关键动作**:
- Frontend: HTML/CSS/JS 基础 + React/Vue 框架 + 手撕算法
- General: 算法/数据结构 + SQL + 系统设计（按岗位选方向）

### 槽位 5：面试准备
**意图**: 准备项目话术（STAR 法则）
**关键动作**: 选 2 个核心项目 → 用 STAR 法则写出话术 → 控制在 3 分钟内
**通用约束**: 强调"讲清楚你做了什么"比"技术多牛"更重要

### 槽位 6：模拟练习
**意图**: 自问自答 / 出声练习
**关键动作**: 准备 5 个高频问题 → 对镜子/录音 → 听回放 → 改进
**通用约束**: 必须包含"出声"动作

### 槽位 7：跟进复盘
**意图**: 投递后不消失，主动跟进
**关键动作**: 3 天无回复发邮件追问 → 内推人打招呼 → 记录每次沟通
**通用约束**: 跟进≠骚扰，保持礼貌和节奏

### 槽位 8：心态复位
**意图**: 防内耗收尾，自嘲解压
**关键动作**: 完成以上 7 步后 → 给自己一个小奖励 → 暂时不想求职
**通用约束**: 必须包含一句自嘲/幽默的话，缓解求职焦虑

## 四、权重分配原则

每槽位 7 条选项，权重固定为 **[10, 9, 8, 7, 5, 3, 2]**：

| 权重 | 定位 | 说明 |
|------|------|------|
| 10 | **默认首推** | 最符合该人格典型行为，大概率被选中 |
| 9 | **高频备选** | 与权重 10 角度相近但措辞不同 |
| 8 | **常用变体** | 换个角度说同一件事 |
| 7 | **均衡选择** | 中规中矩，覆盖另一种常见情况 |
| 5 | **小众角度** | 不那么常规但合理的建议 |
| 3 | **冷门方案** | 适合特定情况的特殊建议 |
| 2 | **彩蛋选项** | 最有梗/最意外的一条，传播友好 |

**关键约束**: 同槽位 7 条选项语义不可重复。可通过以下方式差异化：
- 换句式（祈使 → 反问 → 条件）
- 换角度（技术视角 → 心态视角 → 策略视角）
- 换紧急度（立刻做 → 今天做 → 本周做）
- 换粒度（拆 3 步 → 一步搞定 → 列出清单）

## 五、场景差异

### Frontend（前端开发实习）
- 槽位 4 标题: "前端笔试准备"
- 技术栈提及: React, Vue, JavaScript, CSS, 浏览器原理
- 笔试重点: JS 基础 + 框架原理 + 手撕算法（简单到中等难度）

### General（通用互联网岗）
- 槽位 4 标题: "笔试准备"
- 岗位适配: 后端→算法+SQL、产品→案例分析、数据→SQL+统计
- 笔试重点: 按岗位类型灵活调整，不涉及具体技术栈名

## 六、质量检查清单

生产完成后，每条文案需满足：

- [ ] 至少包含 1 个 `{{变量}}`（推荐 2-3 个）
- [ ] 与同槽位其他 6 条语义不重复
- [ ] 匹配该人格的语气标签（见§二）
- [ ] 匹配该槽位的语义意图（见§三）
- [ ] 渲染后读起来是自然的中文句子（不是拼凑的关键词）
- [ ] 权重在 1-10 范围内
- [ ] 对 `frontend` 场景槽位 4，不要出现后端/数据专用术语
- [ ] 对 `general` 场景槽位 4，不要出现前端专用术语

运行 `npm run validate:content` 自动检查结构 + 变量完整性。

## 七、文案风格字段 `style`（2026-06-26 补充）

`StepOption` 可选字段：

| 值 | 含义 | 内容状态 |
|----|------|----------|
| 省略或 `"casual"` | 松弛自嘲（默认） | ✅ 2688 条均为该风格 |
| `"serious"` | 务实高效 | ❌ 待撰写，见 BACKLOG |

DefuserPage 提供风格切换 UI；在 serious 文案填充前，切换后可能与 casual 相同。合并新文案时须为每条 option 标注 `style`。
